# UI overhaul: navigation restructure, page extraction, filter expansion, and Supabase scaffolding

This change transforms InternHub from a single-page app with profile creation embedded in the homepage hero into a multi-page architecture. The navbar becomes Home, Opportunities, Post, Mail. Profile creation moves to `/get-started`, the homepage hero becomes a marketing card, and a full `/opportunities` page shows all 13 listings with expanded filtering. A contact form modal replaces the previously inert Register/Enroll buttons. Supabase client scaffolding lands but is not yet wired to any page.

**Watch for:** Type badge dark-mode readability is broken — `dark:text-black` on dark backgrounds makes text invisible (confirmed). The salary filter parses only the first number from pay strings, causing annual-salary listings like "$64k-$76k/yr" to match the "$60+/hr" bucket incorrectly (confirmed). The rating filter UI exists but filters nothing because opportunity data has no `rating` field (confirmed). The career path "Learn as you go" card centering depends on whether `justify-center` on a half-width flex item meets the design intent (likely still visually off).

## High-level view

Profile creation at `/get-started` has a 195-country dropdown and commute-radius selector, but the data stays in local component state — closing the tab loses the profile entirely, and Supabase is not wired in.

The opportunity data is duplicated verbatim between `app/page.tsx` and `app/opportunities/page.tsx` — 13 hardcoded objects and the `Opportunity` type appear in both files.

The filter system introduces salary range, working hours, and rating categories, but the mappings are approximate. "Full-time" maps to `type === "In person"`, "Part-time" maps to `level === "Internship"`, and the rating filter has no data to operate on.

The type badge color fix forces `text-black dark:text-black` everywhere. Light mode looks correct, but dark-mode badges become unreadable.

The contact form modal shows "Submitted successfully!" but discards all form data — no API call, no Supabase insert, no email dispatch.

<details>
<summary>Issues (7)</summary>

1. **Dark-mode badge text invisible** — Type badges use `dark:text-black` which is unreadable on dark backgrounds. Change to `dark:text-white` or a light color variant like `dark:text-violet-100`.
2. **Salary filter misclassifies annual pay** — `parseFloat(job.pay.replace(/[^0-9.]/g, ""))` extracts `64` from "$64k-$76k/yr" and matches it to the "$60+/hr" bucket. Normalize pay to a common unit before comparing, or add separate range buckets for annual salaries.
3. **Rating filter is non-functional** — The star-rating UI sets `minRating` state but the filter function never references it. Either add a `rating` field to opportunity data or remove the filter to avoid misleading users.
4. **"Learn as you go" card centering** — The flex-wrap layout gives all cards `w-[calc(50%-3px)]`, so the 5th card is a half-width box on the last row. With `justify-center` it floats in the center but isn't full-width centered. Consider CSS grid with explicit last-child placement.
5. **Opportunity data duplicated across two files** — The 13-item `opportunities` array and its `Opportunity` type are copy-pasted between `app/page.tsx` and `app/opportunities/page.tsx`. Extract to a shared module.
6. **Profile form has no persistence** — The `/get-started` form stores data in React state only. Navigating away loses it. Until Supabase integration is wired, persist to `localStorage` or `sessionStorage`.
7. **ESLint config relaxes rules broadly** — Turning off `no-explicit-any` and `no-unescaped-entities` project-wide removes useful guardrails. Prefer per-file `eslint-disable` comments for specific known exceptions.

## Dark-mode type badge visibility

The original badges used `text-violet-700 dark:text-violet-300`. The new implementation forces `text-black dark:text-black`:

```tsx
Remote: "bg-violet-100 text-black dark:bg-violet-900/40 dark:text-black",
"In person": "bg-blue-100 text-black dark:bg-blue-900/40 dark:text-black",
Hybrid: "bg-teal-100 text-black dark:bg-teal-900/40 dark:text-black",
```

On `dark:bg-violet-900/40` (a dark purple at 40% opacity over the card's dark surface), black text has near-zero contrast. This appears in both `app/page.tsx` and `app/opportunities/page.tsx`. The dark-mode side needs `dark:text-violet-100` or similar.

## Salary filter parsing treats all numbers as hourly

```tsx
const payNum = parseFloat(job.pay.replace(/[^0-9.]/g, ""));
if (range === "$60+/hr") return payNum > 60;
```

For "$64k-$76k/yr", `parseFloat` extracts `64` (the 'k' is stripped), which matches the `$60+/hr` bucket. But $64k/year is ~$31/hr and should match `$20-40/hr`. The "Junior Backend Developer" at "$55k-$70k/yr" has the same problem. Either normalize to hourly equivalents or detect the `/yr` suffix.

## Career path grid centering

The section changed from CSS-grid with `gap-px` to flex-wrap with `gap-[5px]`:

```tsx
<div className="mt-12 flex flex-wrap justify-center gap-[5px]">
```

Each card is `w-[calc(50%-3px)]`. With 5 cards, the 5th wraps to a third row. `justify-center` does center it within the flex line, but the card is still only half the container width — it's a narrow box floating in the middle rather than a prominently centered element. The 5px gap requirement is met, but the centering may not match the intended visual weight. A CSS grid with the last child spanning or explicit `justify-self-center` would give more control.

## Duplicated opportunity data

Both pages define identical 13-item arrays and an identical type. Adding an opportunity or changing a pay rate requires synchronized edits. A `lib/data/opportunities.ts` shared module would eliminate this and later serve as the seam where Supabase queries replace static data.

## Contact form modal discards submissions

The form collects email, phone, and message, then sets `contactSubmitted = true` and shows "Submitted successfully! We will reach out to you shortly." The data is never sent anywhere. The success copy makes a promise the system cannot keep.

## Supabase client will need @supabase/ssr for App Router

The current `createClient()` uses `@supabase/supabase-js` directly. When actual data fetching is added, the App Router's server component and route handler patterns require `@supabase/ssr`'s `createBrowserClient` / `createServerClient` for proper cookie-based auth. The current scaffolding will need replacement rather than extension.

## ESLint rule relaxation

Turning off `no-explicit-any` globally matters as Supabase integration grows — the `any` type defeats the purpose of the typed database interfaces in `lib/types/database.ts`. Narrower per-file disables would preserve the guardrail for new code.
