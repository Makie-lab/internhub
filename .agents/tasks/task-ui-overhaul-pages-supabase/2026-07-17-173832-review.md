# UI overhaul: navigation, pages, filters, and Supabase scaffolding (v2)

This change restructures InternHub into a multi-page app with a new navigation (Home, Opportunities, Post, Mail), dedicated profile creation at `/get-started`, a full `/opportunities` browser, a mail-links page, expanded filters (salary range, working hours), shared opportunity data, a contact-info modal for Register/Enroll, and Supabase client scaffolding. The v1 review's five actionable issues (dark-mode badges, salary filter parsing, non-functional rating filter, data duplication, ESLint relaxation) have been addressed.

**Watch for:** The working hours filter uses proxy semantics that will mislead users — "Full-time" matches only "In person" jobs, "Part-time" matches only "Internship" level (confirmed). The contact form shows a success message but discards all submitted data with no backend call (confirmed). The `parsePayToHourly` sort comparator for "Highest pay" still uses string `localeCompare` rather than the numeric utility it now has (confirmed).

## High-level view

The working hours filter label/behavior mismatch is the most user-facing gap remaining. "Full-time" filters on work-type (In person), and "Part-time" filters on experience level (Internship). A user selecting "Part-time" expects hours-based filtering, not level-based filtering. Since the opportunity data has no explicit hours field, this filter either needs data to back it or should be labeled to match its actual behavior (e.g., "On-site only", "Internships", "Remote/Hybrid").

The "Highest pay" sort uses `localeCompare` on raw pay strings like "$32-$38/hr" and "$64k-$76k/yr". This produces alphabetical ordering, not actual salary ordering. The `parsePayToHourly` utility already exists and could back a numeric sort.

The contact form modal shows a success confirmation without transmitting data anywhere. The user requirement was to "display contact info like gmail, number" when Register/Enroll is clicked. The current implementation asks for the user's contact info rather than displaying the organization's contact info — inverting the intended direction.

1. **Working hours filter semantics are misleading** — "Full-time" maps to `job.type === "In person"` and "Part-time" maps to `job.level === "Internship"`. Relabel the filter options to match actual behavior, or add an explicit `hours` field to opportunity data.

4. **Contact form data is silently discarded** — If the form direction is intentional (collecting visitor info), `submitContactForm` sets a success boolean but never stores or sends the data. Persist to localStorage or wire to a backend.
5. **Supabase `createServerClient` will need replacement for auth** — The current `@supabase/supabase-js` direct client won't handle cookie-based sessions in App Router server components. Flag for future when auth is wired.

## Working hours filter proxies unrelated fields

The filter UI presents "Full-time", "Part-time", and "Flexible" as working-hours categories. The implementation:

```tsx
if (hours === "Full-time") return job.type === "In person";
if (hours === "Part-time") return job.level === "Internship";
if (hours === "Flexible") return job.type === "Remote" || job.type === "Hybrid";
```

"In person" jobs are not exclusively full-time (the Machine Learning Intern is in-person but still an internship with no hours guarantee). "Internship" level is not synonymous with part-time — many internships are 40-hour roles. The "Flexible" mapping to Remote/Hybrid is the only one with a defensible semantic connection.

The opportunity data lacks an explicit hours field, so this filter cannot produce correct results without either enriching the data or relabeling the checkboxes to match what they actually filter (work modality and level, not hours).

## Pay sort uses string comparison

Both `page.tsx` and `opportunities/page.tsx` sort with:

```tsx
return [...result].sort((first, second) =>
  sortOrder === "Highest pay" ? second.pay.localeCompare(first.pay) : 0,
);
```

String sort produces: `$64k-$76k/yr` > `$55k-$70k/yr` > `$50-$60/hr` > `$40-$48/hr` > ... This is accidentally close to correct for hourly rates (leading digit drives it), but the annual salaries sort incorrectly relative to hourly rates ($64k/yr ~ $31/hr should sort below $50-$60/hr). `parsePayToHourly` already exists and would produce a correct numeric sort:

```tsx
return [...result].sort((first, second) =>
  sortOrder === "Highest pay" ? parsePayToHourly(second.pay) - parsePayToHourly(first.pay) : 0,
);
```

## Contact modal direction

The user requirement states: "register and enroll feature not working, should display contact info like gmail, number any information needed." This reads as wanting the modal to show InternHub's own contact information (email address, phone number) so users know how to reach the organization. The current implementation instead presents an input form asking the visitor for their email, phone, and message — then discards it with no API call, no localStorage write, no Supabase insert.

If the intent is truly a contact collection form, it needs a backend or at minimum localStorage persistence. If the intent is to display static contact info, the modal should show InternHub's email and phone with copy-to-clipboard affordances rather than form inputs.
