# Multi-page restructure with shared Navbar, display font, and split filters

InternHub was a single ~500-line page with anchor-link navigation. This change breaks it into four App Router routes (`/`, `/for-employees`, `/for-students`, `/post`), extracts a shared `Navbar` and `Footer` rendered in `layout.tsx`, adds Playfair Display as a second font for headings, and splits the "Certifications & webinars" filter into two independent checkboxes with separate state and filtering logic. A build-blocking `turbopack.root` pointing to a developer's local filesystem was also removed.

**Watch for:** Dark mode initializer calls `localStorage` and `window` at module-level inside `useState`, which will throw during SSR if any route drops `"use client"` or if the Navbar is accidentally imported into a server component (confirmed). The filter split introduces an AND relationship between the two checkboxes — checking both Certifications and Webinars narrows results to only roles that have *both*, which may surprise users expecting a union (confirmed). The `/post` form has no submit handler beyond `preventDefault` — the success message claims submission happened but nothing was persisted (confirmed).

## High-level view

The routing restructure places four `"use client"` leaf routes under `Navbar` and `Footer` in the root layout. Dark mode state lives in the Navbar as client-side localStorage with a `typeof window` guard, which means the server always renders light — users with dark preference see a flash on every cold load.

The filter separation adds `hasWebinar` to the opportunity type and data, introduces `showWebinarOnly` state, and wires it into the `useMemo` filter chain with AND semantics alongside `showCertOnly`. When both are checked, only roles with *both* attributes appear — one role in the sample data qualifies.

The `/post` page renders a detailed form with a success state toggle but no data persistence or API call. The success message is misleading since nothing was submitted.

## SSR boundary and dark mode hydration

The `Navbar` component defines `getInitialDarkMode()` at module scope and passes it as the lazy initializer to `useState`:

```tsx
function getInitialDarkMode(): boolean {
  if (typeof window === "undefined") return false;
  const stored = localStorage.getItem("internhub-dark");
  if (stored !== null) return stored === "true";
  return window.matchMedia("(prefers-color-scheme: dark)").matches;
}
// ...
const [darkMode, setDarkMode] = useState(getInitialDarkMode);
```

The `typeof window` guard means the server always returns `false`, so the initial server-rendered HTML never includes the `dark` class on `<html>`. The `useEffect` then adds it client-side, causing a visible flash for dark-mode users. All four routes are statically generated per the build output, so users see the FOUC on every cold load. (confirmed)

## Filter AND semantics on certification/webinar split

The original single checkbox (`showCertOnly`) filtered to roles with `hasCertification`. The split adds a parallel `showWebinarOnly` filter:

```tsx
const certMatches = !showCertOnly || job.hasCertification === true;
const webinarMatches = !showWebinarOnly || job.hasWebinar === true;
return ... && certMatches && webinarMatches;
```

When both are checked, a role must have *both* `hasCertification` and `hasWebinar` to appear. Only one role in the sample data (Notion's Software Engineering Intern) satisfies both. Most filter UIs treat multiple selections within a category as additive (show roles with certs OR webinars). (confirmed)

## Post page as UI mockup

The `/post` page renders a complete form and a submit button. The submit handler:

```tsx
function handleSubmit(e: FormEvent<HTMLFormElement>) {
  e.preventDefault();
  setSubmitted(true);
}
```

No field values are captured in state. The success screen says "Your opportunity has been submitted" — misleading since nothing was stored or transmitted. (confirmed)

## Footer placeholder links

The `Footer` component links Privacy, Terms, and Help center all to `/`. These ship in the shared layout across all pages and route to the homepage with no useful distinction. (confirmed)
